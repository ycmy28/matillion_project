Inferred Data Flow

Based on the included templates, the GCTS package most likely uses the following operating pattern: 

GCTS source files
  -> raw load table creation
  -> load-layer insert
  -> ETL metadata enrichment
  -> stage-layer load
  -> soft-delete / historization handling

  -> curated downstream outputs

File Handling Pattern

The reusable load-data jobs support both CSV and JSON. The default file type variable in the included template is json, and the flow branches between:

  • CSV from default stage
  • CSV from external stage
  • JSON from external stage

Soft Delete And Historization Logic

The included Partial Delete framework jobs dynamically build SQL to compare source and target records, detect business keys, separate varchar vs non-varchar comparison logic, and then:

  • mark existing records as deleted using _ETL_RECORD_INDICATOR = 'D', or
  • delete and recreate records when load timestamps indicate a newer source version

This confirms that the framework is designed for incremental, history-aware stage maintenance.
