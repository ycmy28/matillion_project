Pipeline Overview

Pipeline Name: orc_master_e2e_dcs_lz_pipeline

Source System: DCS (Digital Commerce System) market-specific staging tables in Snowflake

Target System: Snowflake presentation layer

Data Domain: Consumer identity / user profiles

Deployment Status: DEV ✅   QAS ✅   PRD ❌ (not yet deployed to PROD)

Purpose

This pipeline harmonizes DCS user data from multiple market-specific staging tables into curated integration views and presentation-layer tables. The documented scope covers consumer identity and user profile entities across 14 markets (AU, CA, DK, GR, IL, IT, KZ, LT, NL, NZ, PT, RO, SE, UA).

The pipeline produces:

  • A consolidated user profile table (MDE_DCS_USER)
  • An extended user attributes table (MDE_DCS_USER_EXTRA)
  • A guest user table for Canada market (MDE_DCS_GUEST_USER)
  • A profile field reference table (RDE_DCS_PROFILE_FIELD)
  • Unit-test orchestration jobs for each published object

Common Transformation Pattern

All four DCS Phase 1 transformations follow the same core pattern:

Market-Specific Staging Tables
  │
  ├─► UNION ALL across markets (with schema alignment)
  ├─► Filter - Bad Data Quality
  ├─► Add Dedup Columns (COUNT / ROW_NUMBER)
  ├─► Calculator - column transformations and type casting
  ├─► Derive COUNTRY_CODE from __entity_name
  ├─► Derive VALID_FROM
  ├─► Select Columns
  └─► View creation in integration schema

Shared Transformation Controls

Each transformation uses:

  • Scalar variable: jv_source_table
  • Grid variables: gv_calculator, gv_table_input_column_list

This indicates a metadata-driven pattern where column lists and output mappings are controlled through Matillion grids rather than hard-coded component definitions.

DCS-Specific Transformation Rules

The data model adds several implementation details specific to DCS Phase 1:

  • Source staging tables are per-market under DB_CONS_STG_PRD.STG_DCS
  • All four published objects are modeled as SCD2
  • All four published objects use Load type: DELTA
  • Country code is derived from the staging entity name: UPPER(SPLIT_PART(__entity_name, '_', 2))
  • Every object applies a timestamp quality rule before publication:
    • TRY_TO_TIMESTAMP_NTZ(updated_at) <= __load_ts OR TRY_TO_TIMESTAMP_NTZ(updated_at) IS NULL
  • Every object derives VALID_FROM with the same general rule:
    • Use the business timestamp (updated_at / Updated_Timestamp) when it is present and the row is the surviving duplicate candidate
    • Otherwise fall back to __load_ts
  • Every object uses duplicate handling based on:
    • COUNT(*) OVER (...) AS dup_cnt
    • ROW_NUMBER() OVER (... ORDER BY __load_ts ASC) AS rn_by_load_ts

Multi-Market UNION ALL Pattern

Unlike the Interactions pipeline which reads from single staging tables, DCS Phase 1 unions market-specific tables:

FROM (
    SELECT * FROM DB_CONS_STG_PRD.STG_DCS.T_DCS_AU_<ENTITY>
    UNION ALL
    SELECT * FROM DB_CONS_STG_PRD.STG_DCS.T_DCS_CA_<ENTITY>
    UNION ALL
    ...
    SELECT * FROM DB_CONS_STG_PRD.STG_DCS.T_DCS_UA_<ENTITY>
) i
WHERE
    TRY_TO_TIMESTAMP_NTZ(i.updated_at) <= i.__load_ts
    OR TRY_TO_TIMESTAMP_NTZ(i.updated_at) IS NULL

Special note for MDE_DCS_USER: The UA market has an extra column cc_coms. All other markets add NULL cc_coms for UNION ALL compatibility. UA uses SELECT * EXCLUDE(cc_coms), cc_coms to ensure column ordering.

Historization and Deduplication Logic

The model shows a consistent historization pattern across all DCS Phase 1 entities:

1. UNION ALL market-specific staging tables from STG_DCS
2. Parse business timestamp with TRY_TO_TIMESTAMP_NTZ(updated_at)
3. Exclude records where business timestamp is after ingestion timestamp
4. Partition by object business keys plus business timestamp
5. Compute duplicate count and earliest-load row number
6. Set VALID_FROM to business timestamp when usable
7. Otherwise set VALID_FROM to __load_ts
8. Publish as SCD2-ready output with VALID_FROM / VALID_TO / ACTIVE_FLAG

Table Descriptions

  • MDE_DCS_USER — Main table containing primary user attributes, including identity identifiers, personal profile information, contact details, registration and activity timestamps, consent and opt-in flags, company and store affiliations, and lifecycle status indicators.
  • MDE_DCS_USER_EXTRA — Contains additional and supplementary information related to users, capturing attributes that extend the core user profile, such as activity details, consent and opt-in indicators, marketing preferences, anonymization context, and other user-specific metadata.
  • MDE_DCS_GUEST_USER — Table that holds information from the guests form from Canada market. A guest is not a User, as a Guest has no account or profile.
  • RDE_DCS_PROFILE_FIELD — Table used to link the consumer, device and order with their profile information. Contains key-value style attributes associated with user profiles, such as profile identifiers, field names, field values, and related update timestamps.

Error Handling

The presentation orchestration jobs include explicit failure-handling patterns:

  • Retry - Initial components before the main create/load step
  • End Failure - Presentation table terminal path
  • Send an Email with HTML Body type notification component (to ${ev_harmonization_email_recipient})
  • Delete Transient table 3 cleanup component

Additional controls visible in the export:

  • Unit-test gates validate row-count or test-result expectations
  • Query tagging supports troubleshooting in Snowflake history
  • Task history loading supports operational monitoring

Business Use Cases

  • Build a consolidated consumer identity view across 14 DCS markets
  • Enrich consumer profiles with extended attributes (loyalty, consent, marketing preferences)
  • Support guest form analysis for Canada market
  • Link consumer profiles to devices, orders, and profile field metadata via RDE_DCS_PROFILE_FIELD
  • Enable downstream data products that need historized user identity records
  • Support row-level security (RLS) filtering by market via RDE_BRIDGE_CONSUMER_RLS (separate pipeline)
  • Track consumer lifecycle: registration, activity, deregistration, anonymization

Dependencies

Upstream Dependencies

  • DCS ingestion into STG_DCS (market-specific tables per entity)
  • Snowflake environment variables and database schemas
  • Master-pipeline dispatch logic through DISP Ingestion - CONS 1

Internal Dependencies

  • Integration views must be created before presentation harmonization runs
  • Presentation tables depend on loader metadata grids and primary-key configuration
  • Unit tests depend on successful publication of the presentation tables
  • RDE_BRIDGE_CONSUMER_RLS (separate pipeline) is required for RLS sharing scripts

Maintenance and Operations

Operational Notes

  • Use the query tag ${job_name}-${run_history_id} to trace pipeline execution in Snowflake.
  • Validate task-history records after each run.
  • Review unit-test results before promoting changes.
  • Keep the gv_primary_key_configuration grids aligned with business-key definitions.
  • When adding new markets, ensure the UNION ALL in the integration view is updated and column alignment is verified.
  • Production deployment is pending — currently available in DEV and QAS only.

Change Impact Areas

Changes to the following areas are likely to affect downstream consumers:

  • Staging table structures in STG_DCS (especially when new markets are onboarded)
  • Calculator grid expressions in integration transformations
  • Primary-key configuration in harmonization jobs
  • SCD2 loading behavior in the presentation loader framework
  • Column alignment across markets (e.g., cc_coms pattern for MDE_DCS_USER)
